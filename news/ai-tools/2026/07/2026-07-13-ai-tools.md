### 1. Vercel AI SDK / ai@7.0.19〜7.0.22（MCP ツールドリフト検知 ＋ repairToolCall 昇格）

- 種別: OSS / SDK
- 関連: Vercel AI SDK / MCP
- 公開・更新日: 2026-07-09（`ai@7.0.19`）〜 2026-07-11（`@ai-sdk/groq@4.0.8` / `@ai-sdk/groq@3.0.51`）
- 概要: `ai@7.0.19` が **MCP のツール定義ドリフト検知**を導入。**`fingerprintTools`** は、あるツールセットを信頼した瞬間に**サーバ側が制御するフィールド（説明・入力スキーマ・タイトル）を指紋（チェックサム）として固定**する。**`detectToolDrift`** は、その後に取得したツール定義を指紋と差分比較し、**注入された説明や広げられたスキーマを検出**する。これにより、エージェントが一度信頼したリモート MCP サーバがツールの説明やスキーマを**後から静かに書き換える「rug pull」攻撃**を、モデルへツールが渡る前に捕まえられる。あわせて 7.0.19 はツール名解決のプロトタイプ汚染対策（own-property チェック強化）も含む。続く `ai@7.0.20`（07-10）で **`repairToolCall` オプションが experimental から stable に昇格**（旧 `experimental_repairToolCall` は非推奨エイリアスとして残置）、Express / Next.js での圧縮レスポンスの逐次フラッシュも改善。`7.0.22`（07-10）は Cartesia プロバイダ（Sonic 3.5 音声生成、Ink-Whisper バッチ／Ink 2 リアルタイム文字起こし）を追加。07-11 には `@ai-sdk/groq` 4.0.8 / 3.0.51 が **Groq のプロンプトキャッシュヒットを `cachedInputTokens` に正しく計上**する修正を出した。
- 何が便利か: MCP を組み込むエージェントで、**信頼済みツールの「後出しすり替え」を検知する仕組み**を SDK 標準で回せる。`repairToolCall` の stable 化で、モデルが壊れた／スキーマ不一致のツール呼び出しを返したときに**自動修復してリトライ**する挙動を、実験扱いを気にせず本番投入できる。
- 使いどころ: 外部・サードパーティの MCP サーバをつなぐエージェント（信頼境界のあるツール接続）、ツール呼び出しの失敗が多いモデル・スキーマでの安定化、Groq 経由でキャッシュ課金を最適化したい構成。
- 開発者にとっての価値: 「MCP をつないだが、リモート側の改変をどう監視するか」という運用課題に**一次対策のプリミティブ**が入った。指紋比較は CI やエージェント起動時のガードに組み込みやすい。`repairToolCall` の昇格は、ツール実行の信頼性を上げたい既存エージェントにそのまま効く。
- 注意点: `detectToolDrift` は**差分の検知**を提供するもので、検知後の遮断・再承認フローは実装側の設計次第。`repairToolCall` は旧 `experimental_` 名から移行が必要（非推奨警告が出る）。バージョンは AI SDK 7 系（v6 系利用者は gateway 依存更新の 6.0.223/6.0.224 が該当）。
- 公式リンク: https://github.com/vercel/ai/releases
- 参考リンク: https://github.com/vercel/ai/blob/main/packages/ai/CHANGELOG.md , https://ai-sdk.dev/
- メモ: 「信頼した瞬間に指紋化 → 後で差分検知」という考え方は、MCP に限らずツール供給側を完全には信頼できない構成の一般的な防御パターンとして応用できる。

#### 使用例

```bash
npm install ai@latest
```

```ts
import { fingerprintTools, detectToolDrift } from 'ai'

// 信頼した時点でツール定義を指紋化して保存
const fingerprint = fingerprintTools(tools)

// 後で取得し直したツール定義がすり替わっていないか検査
const drift = detectToolDrift(fingerprint, refetchedTools)
if (drift.length > 0) {
  // 説明やスキーマが変わっている → モデルに渡す前に遮断／再承認へ
}
```

### 2. Hugging Face Transformers / v5.13.1（vLLM 互換パッチ ＋ 統一エクスポート）

- 種別: OSS ライブラリ
- 関連: Hugging Face / ローカルLLM
- 公開・更新日: 2026-07-11（v5.13.1、パッチ） / 2026-07-03（v5.13.0、本体）
- 概要: v5.13.1 は**最新 vLLM との互換性を回復するためのパッチ**リリース。`remap_legacy_layer_types` をカスタムモデルに対してより防御的にし、**新しい線形レイヤ型名を知らない既存のカスタムコードが壊れる問題**、`_LazyAutoMapping.register` に文字列キーが渡されるケースを修正した。土台となる v5.13.0（07-03）は、**レイヤ宣言・マスク／キャッシュ構築・ハイブリッドアテンション処理を標準化**し、**ONNX / `torch.export` / ExecuTorch にまたがる統一エクスポート**と複数モデルでの fullgraph コンパイル対応を進めたメジャー更新（KimiK 2.5、MiMo-V2-Flash、Qwen3 ASR、VideoPrism など新モデル系統も追加）。
- 何が便利か: v5.13.0 で入った**単一のエクスポート経路（ONNX / torch.export / ExecuTorch）**により、学習済みモデルを推論ランタイムへ持ち出す手順が揃えやすくなる。v5.13.1 で **vLLM 最新版と Transformers を組み合わせたときの破綻を回避**でき、自前サービングの更新が安全になる。
- 使いどころ: Transformers のモデルを ONNX / ExecuTorch へ書き出してエッジ・オンデバイス推論に載せる、vLLM で最新の Transformers を使ってサービングする、カスタムモデル定義を保持したままアップグレードする構成。
- 開発者にとっての価値: 「学習は Transformers、推論は別ランタイム」という定番構成で、**エクスポートの標準化**と**vLLM 互換の回復**という実務のつまずきどころに効く。カスタムモデルを持つチームはレイヤ型名変更の影響を v5.13.1 で吸収できる。
- 注意点: 統一エクスポートの本体機能は v5.13.0（07-03）で、24〜48h の基準からは外れる。今回の在圏分は v5.13.1（07-11）の互換パッチ。v5.13.0 には Gemma 3/4 のスライディングウィンドウ境界のマスク修正など**破壊的変更**が含まれるため、アップグレード時は自前のカスタムモデル・アテンション実装への影響を確認する。
- 公式リンク: https://github.com/huggingface/transformers/releases/tag/v5.13.1
- 参考リンク: https://github.com/huggingface/transformers/releases/tag/v5.13.0
- メモ: vLLM を最新へ上げるタイミングで Transformers も v5.13.1 以上に揃えると、レイヤ型名まわりの非互換を踏みにくい。

#### 使用例

```bash
pip install -U "transformers>=5.13.1"
# 最新 vLLM と組み合わせる場合はこのパッチ以上に揃える
```

### 3. Devin Desktop（旧 Windsurf）/ v3.4.22（Devin Local のエディタ文脈認識 ＋ /mcp）

- 種別: IDE / エージェント
- 関連: Cognition Devin / Windsurf / MCP
- 公開・更新日: 2026-07-04（Devin Desktop v3.4.22）
- 概要: Windsurf は **Devin Desktop へ改称**（Cognition 傘下）。v3.4.22 で **Devin Local が、エディタで開いているファイルを文脈として認識**するようになった。**`/mcp` スラッシュコマンドと、MCP サーバの稼働状況を映すライブ状態パネル**を追加。MCP ツールの権限要求時に**サーバ単位の一括承認（今セッション限り／恒久）**という選択肢が増え、**「Always Allow」の付与がセッションをまたいで永続**するようになった。あわせて、Devin Cloud セッションのネットワーク復帰時の**自動再接続**、自律モードの編集が**レビュー可能な差分**として出るように、`/usage` での ACU 使用量表示、Devin Local を拡張する**プラグイン機構（プレビュー、エンタープライズ向けオプトイン）**も追加された。
- 何が便利か: ローカルエージェント（Devin Local）が**開いているファイルを見て動く**ため、指示なしでも編集対象の文脈を拾いやすい。**MCP 権限をサーバ単位でまとめて付与・永続化**でき、ツールごとに毎回承認する煩わしさが減る。自律編集が差分で出るのでレビューしやすい。
- 使いどころ: ローカルのコードベースでエージェントを常用する開発、複数 MCP ツールを接続した環境での権限運用、クラウドセッションを長時間走らせる作業（切断からの自動復帰）。
- 開発者にとっての価値: MCP を多用するほど効く**権限のまとめ承認＋永続化**と、`/mcp` によるサーバ状態の可視化は、エージェント運用の摩擦を下げる実務改善。エディタ文脈認識は「今いじっているところ」を起点にした作業と相性が良い。
- 注意点: 07-04 リリースで、24〜48h の基準からはやや外れる境界の採用（前日までの AI ツール記事で未掲載のため今回採用）。Windsurf → Devin Desktop の**改称**が進行中で、ドキュメント・配布物の名称が混在する時期。プラグイン機構はプレビューかつエンタープライズ向けオプトイン。
- 公式リンク: https://devin.ai/desktop
- 参考リンク: https://docs.devin.ai/desktop/changelog , https://devin.ai/blog/windsurf-is-now-devin-desktop
- メモ: MCP 権限の「サーバ単位・恒久」承認は便利な反面、信頼できるサーバに限って使うのが基本。前掲の Vercel AI SDK のツールドリフト検知と発想は補完的。

### Vercel AI SDK（`detectToolDrift` ＋ `repairToolCall`）

選定理由:

- **MCP を組み込むエージェントの「信頼済みツールの後出しすり替え」対策**という、今まさに需要が高い課題に一次対策のプリミティブが入った。
- `repairToolCall` の **stable 昇格（07-10）で本番投入しやすく**なり、`@ai-sdk/groq` の 07-11 修正まで含めて基準期間内の更新が続いている。
- `npm install ai@latest` と数行のコードで試せ、**既存の AI SDK エージェントにそのまま差し込める**導入コストの低さ。

最初に試すこと:

```bash
npm install ai@latest
```

```ts
import { fingerprintTools, detectToolDrift, experimental_createMCPClient } from 'ai'

// 1) MCP からツールを取得し、信頼した時点で指紋化
const fp = fingerprintTools(tools)
// 2) 再取得したツールに対してドリフトを検査（CI／起動時ガードに)
const drift = detectToolDrift(fp, await refetchTools())
if (drift.length) console.warn('tool definition drift detected:', drift)
```
