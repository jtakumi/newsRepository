### 1. GPT-5.6 Sol / Terra / Luna（一般提供）

- 種別: モデル / API
- 関連: OpenAI / Codex / ChatGPT
- 公開・更新日: 2026-07-09（06-26 プレビュー → 07-09 GA）
- 概要: OpenAI の次世代モデル群 GPT-5.6 Sol・Terra・Luna が一般提供に。ChatGPT / Codex / ChatGPT Work / OpenAI API で利用可能。コーディング系では **Terminal-Bench 2.1（計画・反復・ツール連携を要するコマンドライン作業）で新 SOTA**。標準モードの base Sol が 88.8% で GPT-5.5（88.0%）・Fable 5（83.4%）を上回り、上位の Sol Ultra は 91.9%。
- 何が便利か: ターミナル上での多段作業（ビルド→エラー→修正の反復、ツール呼び出しの調整）に強く、Codex CLI や自作エージェントの実行モデルとしてそのまま差し替えられる。
- 使いどころ: Codex CLI / API 経由でのエージェント型コーディング、CI 内の自動修正、シェル操作を伴うタスク自動化。
- 開発者にとっての価値: コマンドライン中心のワークフローで最上位クラスの実行精度を、既存の OpenAI 連携のまま得られる。前世代（GPT-5.5）や Fable 5 からの置き換え検討材料になる。
- 注意点: ベンチ値は公開情報ベースで、実タスクでの体感は要検証。Ultra は上位ティア想定。料金・レート制限はプラン／モデル別に異なる。
- 公式リンク: https://openai.com/index/previewing-gpt-5-6-sol/
- 参考リンク: https://developers.openai.com/codex/changelog
- メモ: 07-09 のAIツールニュース（前日記事）では Bedrock 提供時点で「モデル寄り」として見送っていたが、今回は API/Codex/ChatGPT への正式 GA という別イベントのため採用。

#### 使用例

```bash
# Codex CLI / API で実行モデルを Sol に切り替え
codex --model gpt-5.6-sol
# API では gpt-5.6-sol または別名 gpt-5.6 を指定
```

### 2. Grok 4.5（SpaceXAI × Cursor）

- 種別: モデル / API
- 関連: SpaceXAI / Cursor
- 公開・更新日: 2026-07-09（07-08 に Axios が先行報道、07-09 公開）
- 概要: SpaceXAI がコーディング・エージェント・知識作業向けの最新モデル Grok 4.5 を公開。**Cursor の実エージェント操作トークン（マルチターン編集・大規模リポジトリ文脈・エラー回復）で学習**した MoE モデルで、「Opus クラスをより速く・安く」を掲げる。Cursor、Grok Build、SpaceXAI コンソールで利用可能。
- 何が便利か: 静的なコード集ではなく実際のエージェント作業ログで訓練されているため、複数ファイルにまたがる修正や失敗からの立て直しといった実運用に近い挙動が期待できる。エージェント型ツール利用のスコアが高い。
- 使いどころ: Cursor でのエージェント編集、大規模リポジトリの改修、ツール連携を多用する自動化タスク。低コストで回したい反復作業。
- 開発者にとっての価値: **API 料金が $2 / $6 per Mtok**（入力/出力）、500K コンテキスト、reasoning 設定可と、上位モデルの中では価格・文脈長のバランスが良く、コスト最適化の選択肢が増える。
- 注意点: **EU では未提供**。サーバーサイドツール（web / X 検索・コード実行）は 1,000 回あたり $5 の別課金。ベンチ順位は第三者集計（Artificial Analysis）ベース。
- 公式リンク: https://x.ai/news/grok-4-5
- 参考リンク: https://cursor.com/blog/grok-4-5
- メモ: Cursor と共同で出す初のモデル。VentureBeat は「競合の約半額」と価格面を強調。

### 3. Claude Code / v2.1.205 と Sonnet 5 既定化

- 種別: CLI / モデル
- 関連: Anthropic / Claude Code
- 公開・更新日: 2026-07-08（v2.1.205）
- 概要: v2.1.205 は auto モードに**セッション記録（トランスクリプト）の改ざんをブロックするルール**を追加。`--json-schema` が無効スキーマ時に非構造化出力を無言で返す不具合や、`--max-turns` 到達時に作業中メッセージが失われる不具合を修正。**Windows の worktree 削除で NTFS ジャンクション／シンボリックリンク経由に worktree 外のファイルが消える危険**を修正。エージェント一覧で PR を編集・マージ・コメント・push したセッションが該当 PR をリンク表示するように。あわせて **Sonnet 5 が Claude Code の既定モデル**（ネイティブ 1M トークン、8/31 まで $2/$10 per Mtok の促進価格）。`/doctor` はセットアップ総点検＋自動修復に。
- 何が便利か: 権限・記録まわりの安全性が上がり、`--json-schema` を使った構造化出力の信頼性が改善。既定モデルが 1M コンテキストの Sonnet 5 になり、長文脈タスクを設定なしで扱える。
- 使いどころ: headless / バックグラウンドでの構造化出力パイプライン、Windows での worktree 運用、PR を伴うエージェント作業の追跡。
- 開発者にとっての価値: 記録改ざん防止と worktree の事故修正で、無人・長時間運用の信頼性がさらに向上。`/doctor` で環境不備を自己診断できる。
- 注意点: 促進価格は 8/31 まで。恩恵を受けるには CLI を最新へ更新する必要がある。前日記事は 2.1.203/2.1.204 まで扱ったため、本記事は 2.1.205 と Sonnet 5 既定化が新規分。
- 公式リンク: https://github.com/anthropics/claude-code/blob/main/CHANGELOG.md
- 参考リンク: https://support.claude.com/en/articles/12138966-release-notes
- メモ: `--json-schema` の修正は、構造化出力を前提にしたスクリプト連携に直接効く。

#### 使用例

```bash
npm install -g @anthropic-ai/claude-code
claude --version   # 2.1.205 を確認
claude /doctor     # セットアップ総点検・自動修復
```

### 4. JetBrains Central / AI for Teams and Organizations

- 種別: プラットフォーム / CLI
- 関連: JetBrains（Claude Code / Codex / Gemini CLI 連携）
- 公開・更新日: 2026-07-07 発表（07-08 以降に各紙が詳報、7〜8月に順次提供）
- 概要: JetBrains が、開発者が既に使う AI コーディングツールの「上」に載るガバナンス層を発表。**JetBrains Central CLI** が Claude Code・Codex・Gemini CLI などバラバラのワークフローを同一の組織環境にまとめ、統制・可視化・分析を提供。外部ツールは **MCP**、外部エージェントは **ACP（Agent Client Protocol）**で接続するベンダー非依存設計。共有コンテキストと再利用可能なエージェントワークフロー、コスト管理を備える。
- 何が便利か: チームが好きな CLI を使い続けたまま、利用状況・コスト・ポリシーを組織横断で把握できる。ツール乱立の「見えない AI 利用」を一元管理できる。
- 使いどころ: 複数の AI コーディングツールが混在するチーム／組織でのガバナンス、コスト可視化、共通ワークフローの標準化。
- 開発者にとっての価値: 個々のエージェント CLI を捨てずに、組織要件（監査・統制・分析）を後付けできる。MCP/ACP 準拠なので既存資産を活かしやすい。
- 注意点: 提供は 7〜8月にビジネス顧客へ段階展開。現時点は発表・順次ロールアウト段階で、全機能が即日使えるわけではない。個人開発より組織向けの色が濃い。
- 公式リンク: https://blog.jetbrains.com/blog/2026/07/07/jetbrains-ai-for-teams-and-organizations-from-fragmented-ai-usage-to-coordinated-software-development/
- 参考リンク: https://thenewstack.io/jetbrains-ai-team-governance/
- メモ: 「より良い IDE」ではなく「エージェント群の統制レイヤ」に舵を切った点が論点。MCP/ACP 対応は自作ツール接続の指針にもなる。
