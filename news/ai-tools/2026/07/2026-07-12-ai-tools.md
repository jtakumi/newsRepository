### 1. Claude Code / v2.1.207

- 種別: CLI
- 関連: Anthropic / Claude Code
- 公開・更新日: 2026-07-10（v2.1.207）
- 概要: **Bedrock / Vertex AI / Foundry 上でオプトインフラグなしに Auto モードが有効**になり、これらプラットフォームの**既定モデルが Claude Opus 4.8** へ変更。UX 面では**非常に長いリスト・表・段落・コードブロックのストリーミング時にターミナルが凍結し、キー入力が遅延する不具合を修正**。セキュリティ面では、非対話実行時にリモート管理設定を「同意済み」として恒久記録してしまい**セキュリティ同意ダイアログを出さない不具合**、システム生成の更新に対する**誤ったプロンプトインジェクション警告**、`~/.local/bin/claude` のカスタムランチャースクリプトを自動アップデータが上書きする不具合を修正。応答完了時のトランスクリプトのジャンプ、git worktree・エージェントチームのクラッシュループも修正した。
- 何が便利か: クラウドプラットフォーム（Bedrock / Vertex / Foundry）で追加設定なしに Auto モードと Opus 4.8 が使えるため、エンタープライズ環境での導入が一段軽くなる。長い出力でのフリーズ解消は、大量のログや大きな差分を扱うエージェント作業の体感安定性に直結。
- 使いどころ: AWS Bedrock / Google Vertex / Microsoft Foundry 経由で Claude Code を回す構成、長大なコード生成・ログ出力を伴うタスク、CI やスケジュール実行での無人運用。

- 注意点: 前日（07-11）記事は v2.1.206 までを扱ったため、本記事は v2.1.207 が新規分。Auto モードの既定モデル変更は Bedrock / Vertex / Foundry 上での挙動である点を理解して使う。恩恵を受けるには CLI を最新へ更新する必要がある。
- 公式リンク: https://github.com/anthropics/claude-code/blob/main/CHANGELOG.md

- メモ: セキュリティ同意ダイアログの修正は、非対話（headless）実行で管理設定を扱う環境では特に確認しておきたい変更。

### 2. Cursor / 3.11（Side Chats）

- 種別: Webサービス / IDE
- 関連: Cursor
- 公開・更新日: 2026-07-10（Cursor 3.11）
- 概要: メインのエージェント会話と並走する **Side Chats（サイドチャット）** を導入。`/side`・`/btw`・チャットパネル上部の＋ボタンで、**メインチャットの文脈を引き継いだ派生スレッド**を作れる。各サイドチャットは後から追いかけ・再訪でき、`@`メンションでその内容をメインスレッドへ引き戻せる、独立した永続的なエージェント会話。既定では読み取り・検索・回答に寄せた挙動で、メインのエージェントを走らせたまま「疑問点の確認」「別案の調査」「意思決定のサニティチェック」を安全に行える。あわせて**エージェントのトランスクリプト（過去の会話）を検索**でき、名前や PR 番号を超えた本文検索で過去のやり取りを素早く辿れる。プロジェクト／リポジトリのピッカーも簡素化された。
- 何が便利か: メインのコード変更を止めずに「これ何だっけ？」「この方針で合ってる？」を横で確認でき、コンテキストを汚さずに調べ物ができる。過去チャットの本文検索で「あのとき何をやったか」を掘り起こしやすい。
- 使いどころ: 長時間走らせるエージェントタスクの途中確認、方針を決める前の代替案リサーチ、レビュー中のサニティチェック、過去セッションの実装意図の掘り起こし。
- 開発者にとっての価値: 「メインの作業スレッドを一本に保ちつつ、脱線を別レーンに逃がす」という会話管理は、複数の関心事が交錯するエージェント開発の実務課題に直接効く。トランスクリプト検索は履歴が増えるほど価値が上がる。
- 注意点: Cursor は GUI エディタのため、機能は Cursor 本体の更新後に利用可能。サイドチャットは既定で読み取り・検索寄りの挙動である点を踏まえ、書き込みを伴う作業はメインスレッド側で扱うのが基本。
- 公式リンク: https://cursor.com/changelog
- 参考リンク: https://releasebot.io/updates/cursor
- メモ: `/side` と `/btw` はどちらもサイドチャット生成のトリガー。メインを止めたくない「ながら調査」に向く。

### 3. Anthropic API キーの有効期限設定 / Claude Console

- 種別: API / ドキュメント更新
- 関連: Anthropic / Claude
- 公開・更新日: 2026-07-11
- 概要: Claude Console の API キー作成時に**有効期限を選択できる**ようになった。通常の API キー・Admin API キーとも、**プリセット（3時間 / 1日 / 7日 / 30日）・カスタム期間・Never** から選べる。組織に最大有効期限ポリシーがある場合、プリセットとカスタム期間はその上限に制限され、Never は選べない。期限が近づくと**キー作成者へメール通知**（有効期間14日以上のキーは7日前、7日以上のキーは1日前。それより短命なキーは警告なしで失効）。**失効後のキーは 401 `authentication_error` を返し、再有効化は不可**（新規キーの作成が必要）。Console のキー一覧に各キーの有効期限が表示され、Admin API の List / Get API Keys エンドポイントが `expires_at` を返すため、監査・ローテーションに使える。
- 何が便利か: 短期の検証用キーやデモ用キーを「自動で失効する」形で発行でき、消し忘れによる漏洩リスクを構造的に下げられる。Admin API の `expires_at` で棚卸しを自動化できる。
- 使いどころ: CI／一時環境向けの短命キー発行、外部委託・PoC で配る期限付きキー、シークレットマネージャで管理する長期キー（Never）と使い分けたローテーション運用。
- 開発者にとっての価値: キー管理の基本作法（最小権限・期限付き・定期ローテーション）を Console 標準機能で回せる。`expires_at` を監査スクリプトに組み込めば、失効前アラートを自前で作れる。
- 注意点: 失効キーは復活できないため、本番用の長期キーはシークレットマネージャ管理の Never＋自前ローテーションが基本。短命キー（7日未満）は警告メールが飛ばない点に注意。組織ポリシーで最大有効期限が設定されている場合は Never を選べない。
- 公式リンク: https://platform.claude.com/docs/en/manage-claude/authentication
- 参考リンク: https://releasebot.io/updates/anthropic
- メモ: Admin API の List / Get API Keys が返す `expires_at` を使えば、失効前の自動アラートやローテーションを CI に組み込める。

#### 使用例

```bash
# Admin API で各キーの expires_at を確認し、失効前のキーを洗い出す
curl https://api.anthropic.com/v1/organizations/api_keys \
  -H "x-api-key: $ANTHROPIC_ADMIN_KEY" \
  -H "anthropic-version: 2023-06-01"
# レスポンスの各キーの expires_at を見て、期限が近いものをローテーション
```
