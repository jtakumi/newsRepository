### 1. LiteLLM / v1.92.0（MCP OAuth 正式対応 ＋ Gemini 3.5 Flash Day-0）

- 種別: OSS / API ゲートウェイ（プロキシ）
- 関連: LiteLLM / MCP / Cursor / Gemini
- 公開・更新日: 2026-07-12（v1.92.0、安定版）
- 概要: 多数の LLM プロバイダーを OpenAI 互換 API に集約する LiteLLM の安定版 v1.92.0 が、**MCP の OAuth 認証をネイティブサポート**。とくに **Cursor 向けの MCP OAuth** に対応し、**OAuth 認証済みの MCP サーバーに対してツール一覧の取得・ツール呼び出しを UI から**実行できるようになった。あわせて **Google Gemini 3.5 Flash の Day-0 対応**（managed agents 対応、gemini-3.1-flash-lite のコストマッピング追加を含む）、**Microsoft Purview DLP ガードレール**連携、Interactions API を Google の 2026-05 スキーマへ移行、Redis カウンタのシード改善（複数ポッドでの二重シード防止）などを追加。配布 Docker イメージは cosign で署名検証される。
- 何が便利か: 自前の MCP サーバーを OAuth 保護したまま、LiteLLM を**認証つきの MCP 接続ハブ**として使える。Gemini 3.5 Flash を追加設定なしにその日から呼べ、Purview DLP を通せば企業の情報漏えい対策ガードレールを LLM 経由の入出力に噛ませられる。
- 使いどころ: 複数プロバイダー・複数 MCP を 1 つのプロキシで束ねるチーム基盤、OAuth 保護された社内 MCP を Cursor などのクライアントから使う構成、DLP を必須とする企業環境での LLM ゲートウェイ。
- 開発者にとっての価値: 「MCP を安全に認証つきで束ねる」という運用課題に、ゲートウェイ側の標準機能で対応できる。Gemini 3.5 Flash の Day-0 対応で、新モデルへの切替をコード変更なし（モデル名指定のみ）で試せる。
- 注意点: MCP OAuth の対応クライアントはまず Cursor が明記されている段階。Purview DLP はエンタープライズ向け（Microsoft 側の設定が前提）。レート制限まわり（ITPM/OTPM の分離など）は v1.93.0-rc 系にも記述があり、安定版で使える範囲はリリースノートで確認するのが安全。
- 公式リンク: https://github.com/BerriAI/litellm/releases/tag/v1.92.0
- 参考リンク: https://docs.litellm.ai/release_notes , https://docs.litellm.ai/docs/mcp
- メモ: LiteLLM を「認証つき MCP のハブ」に据えると、クライアント側（Cursor 等）は LiteLLM だけを向けばよくなり、MCP サーバーごとの資格情報管理を集約できる。

#### 使用例

```bash
pip install -U "litellm[proxy]"
litellm --version          # 1.92.0 を確認
# config.yaml に Gemini 3.5 Flash を登録して proxy 起動
litellm --config config.yaml
```

```yaml
# config.yaml（例）
model_list:
  - model_name: gemini-3.5-flash
    litellm_params:
      model: gemini/gemini-3.5-flash
      api_key: os.environ/GEMINI_API_KEY
```

### 2. Vercel AI SDK / ai@7.0.23〜7.0.26（トレーシング強化 ＋ 文字起こしキャンセル改善）

- 種別: OSS / SDK
- 関連: Vercel AI SDK / xAI（Grok）
- 公開・更新日: 2026-07-13（`ai@7.0.23` / `7.0.25` / `7.0.26`、`@ai-sdk/xai@2.0.77` / `4.0.12`）
- 概要: `ai@7.0.23` が **`embedMany` をトレーシングチャネルのコンテキストで包む**ようになり、埋め込みの一括生成もトレースに乗る。`ai@7.0.26` は**ツール承認後に発生する「孤立したツール呼び出し」を親スパン配下へまとめて可視化**する改善。`ai@7.0.25` は**ストリーミング文字起こしのキャンセル処理を改善**し、非対応モデル時のエラーメッセージを見直した。あわせて `@ai-sdk/xai@2.0.77` が **`grok-4.5` のモデル識別子**を追加、`@ai-sdk/xai@4.0.12` はプロバイダー応答の **URL 検証・資格情報／リダイレクト検証まわりのセキュリティ強化**を含む。
- 何が便利か: エージェントのツール実行・埋め込み・文字起こしといった処理が**トレースの親子関係として整理**され、OpenTelemetry などで追うときに「どのツール承認からどの呼び出しが出たか」を追いやすくなる。長時間の文字起こしを**途中で安全に止められる**のも実務で効く。
- 使いどころ: ツール呼び出しの多いエージェントの可観測性向上、埋め込みを大量に投げる RAG パイプラインのトレース、Grok 4.5 を AI SDK 経由で使う構成、音声文字起こしをストリーミングで扱う UI。
- 開発者にとっての価値: 前バージョン系（07-09〜11 の MCP ツールドリフト検知・`repairToolCall` 昇格）に続く**運用・可観測性まわりの継続改善**。トレースのスパン構造が整うと、本番エージェントの挙動デバッグが楽になる。
- 注意点: 今回分は主にトレーシング・文字起こし・モデル識別子追加の**インクリメンタルな更新**で、大きな新 API ではない。`grok-4.5` の利用は xAI 側のアクセス前提。バージョンは AI SDK 7 系。
- 公式リンク: https://github.com/vercel/ai/releases
- 参考リンク: https://github.com/vercel/ai/blob/main/packages/ai/CHANGELOG.md , https://ai-sdk.dev/
- メモ: 「ツール承認 → 孤立呼び出しを親スパンにまとめる」改善は、承認フローを挟むエージェントのトレースが散らばりがちな問題への実務対応。

#### 使用例

```bash
npm install ai@latest
# grok-4.5 を使う場合
npm install @ai-sdk/xai@latest
```

### 3. Cline CLI / v3.0.40（OAuth プロバイダーへの手動 API キー逃げ道）

- 種別: CLI / OSS
- 関連: Cline
- 公開・更新日: 2026-07-13（CLI v3.0.40）
- 概要: オープンソースのコーディングエージェント Cline の CLI v3.0.40 が、**Cline OAuth プロバイダーに対して手動で API キーを入力できる「エスケープハッチ」**を設定画面に追加。OAuth がうまく通らない環境でも手入力で回避できる。あわせて**モデル切替時にプロバイダー設定がリロードされない不具合**、**シンボリックリンク解決後に Bun のグローバルインストールを自動更新が検出できない不具合**、**トークンリフレッシュ時の一時的なネットワーク／サーバーエラーによる予期せぬログアウト**を修正。ClinePass の使用上限エラーを上限到達時に明確に表示し、**同一セッション内で継続する際にセッション ID を保持**、**コンテキスト圧縮（compaction）の予算処理を堅牢化**した。
- 何が便利か: OAuth 前提のプロバイダー接続でつまずいたときに、**手動キー入力で確実に前へ進める**。Bun でインストールしている場合の自動更新検出や、ネットワーク不調時のログアウトといった「地味に効く」運用不具合が解消。
- 使いどころ: 制限された環境やプロキシ配下で OAuth が通りにくいケース、Bun で Cline CLI を運用している環境、長時間セッションでコンテキスト圧縮を多用するタスク。
- 開発者にとっての価値: エージェント CLI を日常的に回すほど効く**認証・更新・セッション保持の安定化**。手動キーの逃げ道は、認証で詰まって作業が止まる事態を避けられる。
- 注意点: 本リリースは主にバグ修正と認証まわりの改善で、大きな新機能ではない。手動 API キーはあくまで OAuth が使えないときの代替手段。CLI 側の更新なので、GUI（VS Code 拡張）側とはバージョン体系が別。
- 公式リンク: https://github.com/cline/cline/releases
- 参考リンク: https://docs.cline.bot/
- メモ: 直前の GUI 版 v4.0.8（07-11）は GCP Vertex プロバイダーへのモデル追加とカスタムモデル入力に対応。Vertex 経由で非標準モデルを使う構成はそちらもあわせて確認するとよい。

#### 使用例

```bash
npm install -g @cline/cli   # または bun add -g @cline/cli
cline --version             # 3.0.40 を確認
# OAuth が通らない場合は設定から手動 API キーを入力して継続
```

### 4. OpenAI Codex CLI / v0.144.2・v0.144.3（Guardian 自動レビューの回帰ロールバック）

- 種別: CLI
- 関連: OpenAI / Codex
- 公開・更新日: 2026-07-13（v0.144.2 → v0.144.3、いずれも安定版）
- 概要: v0.144.2 が、**Guardian 自動レビューのプロンプト回帰（regression）をロールバック**し、**以前の Guardian 自動レビューのポリシー・リクエスト形式・ツール挙動を復元**した（PR #32672）。直後の v0.144.3 は v0.144.2 以降に取り込んだ変更がない**バージョンのみのリリース**。前日までの記事は v0.144.1（07-09、スタンドアロンインストール不具合の修正）までを扱っていたため、本記事は v0.144.2 以降が新規分。
- 何が便利か: Codex の**自動コードレビュー（Guardian）**を使っている場合、直前に入ったプロンプト回帰で挙動が変わっていた点が元に戻り、レビューの一貫性が回復する。壊れていた期間に更新していた場合は、上げ直すことで従来どおりの挙動に戻せる。
- 使いどころ: Codex の Guardian 自動レビューを CI やローカルのレビューフローに組み込んでいる構成、Codex CLI を最新へ追従させている環境。
- 開発者にとっての価値: 自動レビューは挙動が安定していることが前提の機能で、**回帰を素早く巻き戻した安定版**が出たことで、レビュー結果のブレを避けられる。
- 注意点: 本リリースは新機能ではなく**回帰の巻き戻し**。v0.145.0 系は本記事時点で **alpha（プレリリース）**のため、安定運用は v0.144.3 まで。Guardian を使っていない場合の実利は小さい。
- 公式リンク: https://developers.openai.com/codex/changelog
- 参考リンク: https://github.com/openai/codex/releases/tag/rust-v0.144.2 , https://github.com/openai/codex/releases/tag/rust-v0.144.3
- メモ: 同系統で 07-09 の v0.144.0（承認モード「writes」・対話型認証）〜 v0.144.1（インストール不具合修正）は前日までの記事で扱い済み。今回は v0.144.2 の回帰ロールバックが要点。

#### 使用例

```bash
codex --version            # 0.144.3 を確認（未満なら更新）
# Guardian 自動レビューを使う構成では最新の安定版へ追従しておく
```
